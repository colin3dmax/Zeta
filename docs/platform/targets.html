Zeta 跨平台与运行时架构

状态：draft

更新时间：2026-05-28

适用范围：Zeta target 命名、runtime 分层、panic 策略、allocator contract、build mode 兼容性和 CI matrix。

验收标准：每个 tier-1 target 必须声明 runtime、panic、allocator 和 CI level；WASI、browser、component 必须分开描述。

MVP 边界：target tier、runtime、panic 和 allocator 的取舍以 Zeta MVP Baseline 与冻结边界 为准。

设计目标

  • 统一 Script Mode、AOT Build Mode 和 Component Mode 的 target 选择模型。
  • 明确 desktop、server、mobile、web、embedded 的能力不是等价的，文档必须标注 full、partial、build-only 或 unsupported。
  • 让 core/alloc/std/runtime 分层成为跨平台设计边界，避免平台条件编译散落在业务代码中。
  • 坚持 capability-first：文件、网络、进程、AI 调用等平台能力必须显式声明和授权。

目标平台

  • Desktop / Server：macOS、Linux、Windows，x64、arm64、riscv64。
  • Mobile：iOS arm64、Android arm64，优先以 staticlib/cdylib 进入原生应用。
  • Browser：wasm32-browser，运行在 JS/Web API host 中，不等同于 WASI。
  • WASI：wasm32-wasi，运行在 WASI host 中，文件和网络必须按 WASI capability 描述。
  • WASM Component：wasm32-component，交付 component artifact，通过 WIT imports/exports 和 host 交互。
  • Embedded：arm-none-eabi、riscv32-none，支持 no_std 和可选 alloc。

Panic 与 Allocator Contract

  • unwind：只允许在 native tier-1 且 ABI 明确支持 unwind 的 target 上作为默认策略。
  • abort：WASM、mobile library 和多数 sandbox target 的默认策略，不能跨 host 边界 unwind。
  • trap：no_std 或 board support package 未提供恢复路径时使用，必须在 target adapter 中映射到可观测错误。
  • system allocator：native std target 默认使用系统 allocator，并继承平台 OOM 语义。
  • bundled allocator：browser、WASI 和 component 可随 artifact 打包 allocator，但必须声明内存上限和 OOM 行为。
  • host-provided allocator：component target 可以由 host 注入 allocator，ABI 和 ownership 必须通过 WIT 或等价接口描述。
  • user-provided allocator：embedded target 的 alloc 是可选能力；未声明 allocator 时只能启用 runtime-none 或 runtime-core。

标准库分层

std
  -> alloc
    -> core
  • core：基础类型、Option、Result、panic 抽象，无操作系统、堆、线程、时间和 I/O 依赖。
  • alloc：堆分配、字符串、数组、集合和智能指针，只依赖 allocator contract。
  • std：文件、网络、线程、时间、环境变量、进程、动态库和平台集成。
  • 硬规则：只允许 std -> alloc -> core，禁止反向依赖。

Runtime 裁剪

Runtime 裁剪的目标是让同一门语言能覆盖上层应用和极小系统产物。release 构建必须支持按需链接、dead code elimination、strip/LTO 和显式 runtime tier，不能因为启用了语言工具链就默认带上完整 std、AI provider SDK、agent 框架或调试元数据。

Runtime 依赖 能力 限制
runtime-none core 无 heap、无 OS、无 thread、无 file、无 network。 入口、panic、allocator 由用户或 board support package 提供。
runtime-core core 入口 glue、panic 策略、基础错误处理。 仍不要求 heap。
runtime-alloc alloc allocator contract、String、动态数组、集合、智能指针。 不自动启用文件、网络、线程。
runtime-async alloc future、task、executor 抽象。 事件源由 target adapter 提供。
runtime-std std 文件、网络、线程、时间、环境变量、进程。 mobile 和 wasm 只能标注 partial。

AI 能力扩展边界

AI 能力不是独立 runtime tier。核心内核只保留 capability、schema、tool contract、policy、budget 和 trace 这些可检查契约；provider SDK、agent 框架、模型适配器和高级编排应通过标准库模块或第三方包按需引入。

  • 未导入 AI 模块的程序不得链接 AI provider、agent scheduler、prompt/template 系统或模型客户端。
  • AI 模块即使被导入，也必须通过 capability 授权访问文件、网络、secret 或模型服务。
  • 系统、嵌入式、硬件软件、WASM component 和极小 CLI 可以完全不依赖 AI 相关模块。

质量闸门

  • tier-1 target 必须有明确 panic strategy、allocator contract、runtime tier 和 PR CI level。
  • tier-2 target 可以只做 build-only，但必须能证明 artifact 不依赖未声明的 std、thread、process 或 host import。
  • browser、WASI、component 三类 WASM target 必须分别记录 host capability，不能用一个 wasm 行覆盖。
  • component target 的 CI 以 WIT 接口、adapter 绑定和 host imports 检查为主，不继承 browser 或 WASI 的文件、网络语义。

风险与约束

  • std 如果过早承诺 full cross-platform，会在 browser、mobile、embedded 上失真。
  • AI provider SDK 和 agent 框架不能进入核心 runtime；模型调用必须由外部模块提供，并经过 capability。
  • Script Mode 不适合 embedded 和多数 mobile 发布目标，应优先 host/native。
  • CI matrix 不应一次性全量阻塞 PR，必须区分 PR 必跑和 nightly。
  • 产物体积回归必须可见；release smoke 应记录 artifact size，避免 runtime、std、AI provider SDK 或 agent 框架被隐式链接。
  • allocator contract 必须先定义，否则 embedded、wasm、component 行为会不一致。
  • WASM browser、WASI、WASM Component 是三个不同环境，不应合并成一个能力描述。
  • iOS 动态库、JIT、进程和文件访问限制较多，必须以 partial std 表达。
  • Android 需要明确 NDK、JNI/原生库边界和 target ABI；移动端默认以 library artifact 接入，不承诺完整 runtime-std。
  • panic unwind 在 embedded、wasm、mobile 上不一定成立，target matrix 必须单独列 panic 策略。
