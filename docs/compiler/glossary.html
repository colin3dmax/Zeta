Glossary

把编译器术语翻译成可理解的工程概念。

Zeta 文档会记录语言设计和编译器实现过程。下面的术语表不是学术定义，而是解释这些词在 Zeta 项目里的具体含义、为什么需要它们，以及用户什么时候会看到它们。

从源码到运行结果

一段 Zeta 程序从输入到执行，大致会经过下面这条管线：

Zeta 源码
  -> Lexer：切成 token
  -> Parser：组织成 AST
  -> Resolver：确认名字指向谁
  -> Typechecker：确认类型是否匹配
  -> HIR：更规整的高级中间表示
  -> MIR：更接近执行的中级中间表示
  -> MIR interpreter：直接解释执行 MIR
  -> Backend：未来生成 WASM、机器码或其他目标

核心术语

术语 通俗解释 在 Zeta 里的用途
Lexer 词法分析器。它把源码文本切成更小的词块，例如 fn、main、42、+。 让后续 parser 不必直接处理原始字符串。
Token 源码中的一个词块。比如 return 是关键字 token，answer 是标识符 token。 诊断错误位置、语法高亮和 parser 输入都会使用 token。
Parser 语法分析器。它把 token 组织成树状结构，判断代码是否符合语法。 例如把 40 + 2 识别为“加法表达式”，而不是三个孤立字符。
AST Abstract Syntax Tree，抽象语法树。它是源码语法结构的树状表示。 zeta ast-dump 会输出 AST，方便确认 parser 是否正确理解源码。
Span 源码位置范围。比如一个变量名从第几个字符开始，到第几个字符结束。 错误诊断可以精确指向出错代码，而不是只说“某处错了”。
Resolver 名字解析器。它回答“这个名字指向哪个变量、函数、类型或模块”。 例如检查 answer() 是否存在，import demo.math 是否能找到。
Typechecker 类型检查器。它确认值和操作是否匹配类型规则。 例如 Int 不能当作 Bool 条件，函数参数类型必须匹配。
HIR High-level Intermediate Representation，高级中间表示。它比 AST 更规整，但仍保留较高层的语言结构。 作为后续 MIR 和 backend 的输入基线，减少 backend 直接依赖源码语法细节。
MIR Mid-level Intermediate Representation，中级中间表示。它比 HIR 更接近“机器可以一步步执行的指令”。 zeta mir-dump 输出 MIR，用来检查控制流、局部变量、函数调用和返回路径。
MIR interpreter MIR 解释器。它不生成机器码，而是直接读取 MIR 指令并执行，类似“先把菜谱整理成步骤，再由人按步骤做菜”。 这是 Zeta 早期的第一执行语义基线。还没有完整 WASM/LLVM 后端时，zeta run 先通过它验证程序结果。
Backend 后端。它负责把中间表示变成具体目标能运行的产物，例如 WASM、机器码或其他平台格式。 Zeta 后续会接入 WASM/WASI smoke、LLVM native smoke 等 backend 验证。
WASM WebAssembly，一种可在浏览器和其他运行环境中执行的低层格式。 官网 Playground 加载 zeta.wasm，让浏览器运行真实 Zeta 编译器前端。
WASI WebAssembly System Interface。它给 WASM 程序提供类系统接口，例如文件、时间和标准输入输出。 未来用于验证 Zeta 在命令行/沙箱环境下的 WASM 运行路径。
LLVM 一套成熟的编译器基础设施，很多语言用它生成原生机器码。 Zeta 未来可用 LLVM native smoke 验证 AOT 编译和原生 release 产物。

为什么先做 MIR interpreter

直接生成机器码或 LLVM IR 很诱人，但早期语言还在变化，问题通常出在语法、类型规则、控制流和运行语义上。MIR interpreter 的价值是把问题拆小：

  • 更容易调试：如果 return 40 + 2 没有得到 42，可以直接看 MIR 指令和解释器执行路径。
  • 先定义语言语义：未来无论生成 WASM 还是机器码，都必须和 MIR interpreter 对同一测试输出一致。
  • 降低早期成本：不需要一开始就处理 ABI、寄存器、链接器、平台差异和优化器。
  • 服务 REPL 和 Playground：用户可以更早运行代码，语言团队也能更快验证新语法。

测试和发布术语

术语 解释 例子
Corpus 一组固定测试样例，用来反复验证编译器行为。 testdata/run_struct.zeta、testdata/modules_ok。
Golden test 把某个输出保存成标准答案。以后输出变化时，测试会提醒我们是否破坏了格式或语义。 testdata/core_items.ast 和 testdata/run_enum.mir。
Smoke test 冒烟测试。它不是覆盖所有细节，而是快速确认关键链路没有断。 官网线上 smoke 会确认 WASM 加载、Playground 运行、文档导航和 Feature Tests。
Verifier 验证器。它检查某个中间表示是否内部一致。 MIR verifier 当前会在 mir-dump 和 run 进入执行前检查 local 引用、调用参数、条件类型、return 类型、enum variant、数组元素/下标类型和非 Unit 函数返回路径；后续再扩展到完整控制流图。
Self-hosting 自举或自编译。意思是最终用 Zeta 编写 Zeta 编译器，并让 Zeta 编译器能编译自己。 Stage 0 用 Rust 写冷启动编译器，后续逐步用 Zeta 替换前端模块。

面向用户的命令含义

命令 它做了什么 适合什么时候用
zeta ast-dump file.zeta 只看 parser 是否把源码组织成正确 AST。 调试新语法或理解语法结构。
zeta check file.zeta 执行 parser、resolver 和 typechecker，不运行程序。 确认代码语法、名字和类型是否正确。
zeta mir-dump file.zeta 输出可运行子集的 MIR 文本。 调试运行语义、控制流和函数调用。
zeta run file.zeta 把源码检查通过后 lower 到 MIR，再由 MIR interpreter 执行。 运行无参数 main，查看程序结果。
