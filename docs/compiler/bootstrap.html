Zeta 编译器与自举路线

状态：accepted

更新时间：2026-05-29

适用范围：Zeta Stage 0 到 Stage 3 的编译器实现、自举替换路径、MIR 语义基线和 backend 优先级。

验收标准：Stage 0 能运行 MVP corpus；parser、typechecker、HIR、MIR、backend 和 bootstrap 均有明确质量闸门。

术语说明见 编译器术语解释。如果你不熟悉 AST、HIR、MIR、backend 或 self-hosting，建议先读术语页再看本路线。

自举原则

自编译是最终目标，但第一版编译器必须使用成熟宿主语言实现。Stage 0 主体采用 Rust，以降低早期调试、包管理、测试和 golden corpus 维护成本。

宿主技术栈决策

Stage 0 冻结决策：编译器主体使用 Rust；Zig 不进入当前必需依赖。该决策只覆盖 bootstrap 阶段，不否定后续在平台 glue、allocator/panic adapter 或 cross-linker 中按需引入 Zig。

范围 决策 理由
Stage 0 编译器主体 Rust Cargo、单元测试、快照/golden corpus、错误处理和跨平台 CI 更适合快速建立 lexer、parser、AST、diagnostics、HIR/MIR dump。
CLI 与文档/测试工具 Rust + Python 编译器 CLI 跟随 Rust workspace；轻量仓库检查脚本可以继续使用 Python 标准库，避免引入额外构建依赖。
Zig 暂不引入 当前 Stage 0 不需要 C ABI glue、手写 allocator、panic adapter 或外部 C toolchain 封装；过早引入会增加双工具链维护成本。
Zig 重新评估触发条件 按需评估 当 embedded、WASM/component adapter、Android/iOS native glue、C ABI、allocator/panic contract 或 cross-linker 需求无法用 Rust 工具链清晰覆盖时，再以独立任务评估 Zig。

早期后端顺序

早期语义验证按可调试性优先排序，不把 LLVM 作为唯一真相源。所有 backend 必须通过同一 corpus，行为差异必须先在 MIR 层解释清楚。这里的 MIR 可以理解成“比源码更简单、更接近执行步骤的中间指令”。

  1. MIR interpreter：直接读取并执行 MIR 指令，作为第一语义基线，服务 golden tests、REPL/run 原型和错误定位。
  2. WASM/WASI smoke：验证跨平台、沙箱和浏览器/命令行最小运行路径。
  3. LLVM native smoke：验证原生 AOT、target triple、ABI 和 release 产物。
  4. WASM Component backend：在 tool/plugin/capability sandbox 需求稳定后接入，不阻塞 Stage 0。

决策退出条件

  • 如果 Rust 工具链无法稳定覆盖目标平台 smoke、WASM 产物、CI 或诊断/golden 流程，必须重新评估宿主语言。
  • 如果出现 Android/iOS native glue、C ABI、allocator、panic 或 cross-linker 需求，且 Rust 实现会显著扩大维护成本，可以开启 Zig adapter 评估任务。
  • 如果 LLVM native smoke 与 MIR interpreter 对同一 corpus 行为不一致，以 MIR 语义和规范文档为裁决依据，不能让 backend 反向定义语言语义。

路线目标与非目标

  • 目标：建立可验证、可复现、可逐步替换宿主实现的 self-hosting 路线。
  • 目标：让 Stage 0 成为规范执行器、测试基准和冷启动 seed，而不是一次性脚手架。
  • 目标：每个阶段都有明确输入、输出、退出条件和回归测试。
  • 非目标：Stage 0 不追求完整语言、完整标准库、全平台 backend 或优化编译器。
  • 非目标：自举路径不纳入 async、宏、复杂泛型、完整 borrow checker 和完整 AI agent 语法。

Self-hosting MVP 子集

语言、工具链、runtime 和 AI 能力的统一取舍以 Zeta MVP Baseline 与冻结边界 为准；本节只展开自举路径所需的编译器子集。

类别 必须支持 暂缓
模块 module、import、export、文件级符号。 复杂包可见性、条件编译宏。
语句与表达式 fn、块表达式、let、if/else、while、return、match。 生成器、协程、模式宏。
类型 Bool、整数、String、数组或 slice、struct、enum、Option、Result。 高阶类型、完整 trait specialization、复杂生命周期。
编译器库能力 String、Vec、Map、Set、文件读取、路径处理、诊断输出。 完整 std、网络、线程池、AI provider SDK 或 agent 框架。
错误模型 Result 风格显式错误、错误码、源码 span。 异常作为主错误路径。

阶段定义

阶段 输入 输出 退出条件
Stage 0 Rust 实现的 bootstrap 编译器。 Lexer、parser、AST、diagnostics、name resolution、typechecker、HIR/MIR dump、最小 interpreter 或单一 backend。 能编译并运行 MVP corpus，所有 golden tests 稳定。
Stage 1 Stage 0 编译器和 Zeta 编写的独立前端模块。 优先替换 parser、diagnostics 或 AST dump，不优先替换 typechecker。 Zeta 实现模块与宿主实现对同一 corpus 输出等价。
Stage 2 Zeta 编写的 lexer/parser/AST/HIR/typechecker 核心。 Zeta 编译器可以编译自身核心前端，Stage 0 仍负责 backend 和 runtime seed。 Stage 1 与 Stage 2 编译器在 golden corpus 上输出一致。
Stage 3 完整 Zeta 编译器源码。 Zeta 编译器可编译完整 Zeta 编译器，Stage 0 只作为冷启动 seed。 三轮自举产物稳定，reproducible build、版本 pin 和 bootstrap diff 通过。

编译器管线

Parser、AST、span、错误恢复和 AST golden dump 的第一版冻结规则见 Parser / AST / Span 第一版规格。

Zeta Source
  -> Lexer / Parser
  -> AST
  -> Name Resolution / Type Check
  -> HIR
  -> MIR
  -> Backend Interface
  -> LLVM Native / WASM Component / Interpreter

质量闸门

闸门 验收要求 阻断条件
Parser 语法子集冻结；AST 节点稳定；所有节点带 source span；合法、非法、错误恢复样例有 golden dump。 错误恢复阻塞同文件后续诊断，或 AST dump 不稳定。
Name Resolution / Typechecker name resolution 与 typecheck 分层；支持局部作用域、跨模块符号、函数签名、struct/enum 字段、基础类型推断。 backend 直接依赖未检查 AST，或诊断缺少 span 和错误码。
HIR 去语法糖；绑定 symbol id 和 type id；控制流、调用、字段访问、match/if lower 形式稳定；HIR verifier 通过。 存在悬空 id、类型未附着、非确定性 dump。
MIR 定义 locals、temporaries、call、return、控制流和 drop；当前 MIR verifier 先检查 Stage 0 MIR 的 local 引用、调用参数、条件类型、return 类型、enum variant 和非 Unit 函数返回路径，后续扩展到 CFG、drop/cleanup 和初始化路径。 未定义 local、错误调用参数、错误 return 类型、缺少终结指令或 return path 不完整。
Backend 统一 codegen interface，输入 MIR、target triple、runtime ABI、diagnostic sink；smoke 覆盖整数、字符串、函数调用、模块调用。 backend 直接消费 AST/HIR，或不同 backend 对同一 corpus 行为不一致。
Bootstrap Stage 0 编译 Stage 1，Stage 1 编译 Stage 2，对 corpus 输出一致；三轮自举稳定。 宿主实现长期分叉，或 Zeta 实现无法替换对应宿主模块。

最小 Compiler Corpus

Compiler corpus 是 Stage 0 的回归边界。每个样例必须有稳定输入、预期输出或预期诊断；新增语言特性必须先补 corpus，再进入 parser、typechecker、MIR 或 backend 实现。

类别 最小样例 预期产物 阻断条件
合法语法 空文件、module/import/export、fn、let/let mut、赋值、return、if/else、while、struct、enum、match、函数调用。 AST dump 与 parser golden 文件完全一致；dump 顺序和字段名稳定。 AST dump 非确定性，或新增语法没有 golden 样例。
错误样例 非法 token、缺失分隔符、重复定义、未知名字、不可变赋值、类型不匹配、函数参数数量错误。 诊断包含 code、message、span；同一输入的错误顺序稳定。 诊断缺少错误码/span，或错误恢复导致后续有效 item 无法继续检查。
跨模块 import/export、模块路径、同名符号、不可见符号、跨文件函数调用。 name resolution golden 记录解析成功项和失败诊断；跨模块符号 id 稳定。 resolver 依赖文件遍历顺序，或跨模块错误无法定位到源文件。
IR dump 函数调用、局部变量、赋值、if/while lower、match lower、早返回、基础算术、比较和布尔逻辑。 HIR/MIR dump 可读、可排序、无内存地址；verifier 通过。 dump 包含随机 id，CFG 缺少 terminator，或变量初始化状态不完整。
执行 smoke 无参数 main、整数算术、比较、布尔逻辑、函数调用、let mut、赋值、if/while。 MIR interpreter、WASM/WASI smoke、LLVM native smoke 对同一 corpus 输出一致。 任一 backend 行为与 MIR interpreter 不一致且没有规范化解释。
自举模块 token、span、diagnostic、AST node、parser result、symbol table 的 Zeta 子集实现。 Stage 0 可编译这些模块，并与 Rust 宿主实现的公开 dump/诊断契约一致。 Zeta 子集需要 deferred 特性才能表达编译器核心数据结构。

Golden 测试格式

  • 输入文件放入 testdata/ 或后续 corpus/，按 valid、invalid、ir、run 分组。
  • AST/HIR/MIR golden 使用纯文本，禁止包含内存地址、hash map 原始遍历顺序、系统路径或本地时间。
  • 诊断 golden 至少包含 code、message、file、span.start、span.end；message 可以调整，但 code 和 span 变化必须显式 review。
  • 运行 golden 使用 stdout/stderr/exit code 三元组；backend smoke 必须记录 target 和 runtime tier。
  • CI 至少执行 parser golden、resolver/typecheck 诊断 golden、run smoke 和 docs link check；backend 不稳定时只能降级为非阻断 nightly，不得伪装为通过。

主要风险

  • MVP 子集过大，会让自举路线被 async、宏、复杂泛型和 borrow checker 拖慢。
  • HIR/MIR 边界不清，会导致 backend 反向依赖 AST，后期难以替换。
  • golden dump 不稳定，会让回归测试噪音过大。
  • 标准库能力不足，会导致语言子集能编译但无法编写编译器。
  • Stage 0 没有规格约束，会变成事实标准，Zeta 实现很难追齐。
  • 过早展开多 target，会稀释 self-hosting 主线。
