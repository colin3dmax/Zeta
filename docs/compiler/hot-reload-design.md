# Zeta Hot Reload 设计与不可妥协的性能约束

> 状态基准:2026-06-14。本文定义 Zeta 的"状态保持的运行时热交换"(Erlang/Elixir 式
> hot code swap)如何在 Zeta 的值语义下落地,以及它与"最终性能媲美 C/C++"这一目标如何
> **共存而不互相拖累**。

## 0. 目标

让一个**长跑的 Zeta 程序在不重启、保留累积状态的前提下,替换函数实现**——开发时改一行
`step` 的源码,下一轮迭代就用新逻辑,而状态(分数、世界、计数器……)原样保留。

作为 **Zeta 语言本身的能力**,而非仅 Stage0 工具链的一个 CLI 开关。

## 1. 硬现实(2026-06 核实)

当前 Zeta 与"保留全局状态的热交换"字面冲突,必须先正视:

- 顶层只能声明 `module / import / struct / enum / fn`——**无全局变量、无 static、无模块级
  可变状态**。
- `run` 把 `main()` 跑到结束就返回——**无长跑进程、无事件循环、无"下一帧"概念**。
- **值语义 + copy-on-write**——状态只存在于函数局部。

因此"保留全局状态"无的放矢:没有全局状态可留,也没有进程在跑。要实现该效果,得先造出
**长跑执行模型 + 跨替换存活的状态**,再做替换。

## 2. 与值语义自洽的模型:线程化状态的服务循环

唯一不破坏 Zeta 值语义的"状态保持"方式 = **显式线程化状态**(Elm / Elixir GenServer 同构):

```
fn init() -> State { ... }
fn step(state: State, input: Input) -> State { ... }   // ← 可热替换

// 长跑 driver(由运行时持有,不随替换销毁):
//   state = init();
//   loop { state = step(state, next_input()); 若源变更 → 重新 lower step,下一轮用新版 }
```

**关键**:`state` 由 driver 持有、在迭代间线程化传递;**被替换的是函数表里的 `step`**。
解释器的函数表已是 `HashMap<String, Rc<MirFunction>>`,原子替换一个条目极便宜。新版 `step`
拿到**同一个累积的 `state`**。于是:状态保留 ✅、函数体热替换 ✅、不破坏值语义 ✅、无需引入
全局可变状态 ✅。

状态的"形状"演化(给 State 加字段)属更后期议题,先支持**同形状 State 跨替换**。

## 3. 不可妥协的性能约束(与 C/C++ 目标共存的前提)

C/C++ 的速度来自**静态分发 + 内联 + 无间接层**。热交换的本质要求函数能"换实现"=
**走可替换表(一层间接)**,天然挡住内联与静态分发。两者真实冲突——**但冲突只在"让每次
函数调用都永远走可替换表"时才发生。**

> **约束(不可妥协):热交换只在 opt-in 的粗粒度边界发生(服务循环的 `step` 入口 /
> 显式 `reloadable` 标记);release/native 构建把程序当固定的,做全静态分发 + 内联优化,
> 对热交换付出零成本。语言默认调用语义里绝不引入"每次调用查表"。**

| | 开发 / reload 模式 | release / native 模式 |
|---|---|---|
| 函数分发 | 替换边界走表(可热换) | 静态分发 + 内联(C/C++ 级) |
| 跨函数优化 / 内联 | 替换边界受限 | 全开 |
| 热交换 | ✅ | ❌(程序固定) |

服务循环模型天生适配该约束:替换点是**粗粒度**的(只在 `step` 之间),不是每次调用。即便
开发模式,也只有 `step` 入口间接,`step` **内部调用的一切仍可静态分发**;release 编译一个
固定 `step`,间接层完全消失。

**诚实附注**:
1. "媲美 C/C++"目前 0% 落地——当前整链是**树遍历解释器**,该目标完全依赖**尚未开工的
   native/LLVM 后端**。hot reload 与之正交:既不拉近也不推远。
2. 开发模式下热交换有非零成本(替换边界无法内联)——所有支持热重载的系统(Erlang、JVM
   HotSwap、Unreal Live Coding)都付这个开发态代价;**发布态不付**。

### 3.1 release 能否同时拥有 hot reload 与满性能?

能,且损失小到不可测——前提是两条:

1. **边界粗粒度**:代价的全部来源 = 跨越可替换边界的那次调用无法内联。只有 `step`
   reloadable → 每帧仅一次间接调用(几纳秒,真实负载下测不出)。让**每个**函数都可换
   (细粒度)→ 每次调用都间接、都不能内联 → **全面变慢**。这是红线。
2. **swap 时把新代码编译成 native,而非回退解释**:热交换 ≠ 解释新代码。swap 时即时编译
   新 `step` 为机器码、原子重指函数指针(Unreal Live Coding / .NET Hot Reload 同法)。新代码
   满速 native,只有边界那一跳是间接的;`step` 内部热路径完全优化。

| 对比 | 差距 |
|---|---|
| reloadable-release vs 冻结 release | 仅"少内联一个边界函数 + 每帧一次间接调用",真实负载测不出 |
| reloadable-release vs 解释器 | 快几个数量级(都是 native) |
| 细粒度全可换 vs 冻结 | 明显变慢(到处间接)——禁止 |

严格的"字面零损失"做不到(冻结版能把 `step` 内联进主循环跨边界优化),但差距 = 一次
间接调用/帧。要压到字面零可用 **call-site patching / trampoline**(swap 时改写调用点机器码),
属高级优化,后置;间接调用方案已足够"不降低"。

## 4. 切片路线

- **Slice 1**(进行中):持久运行时 + 迭代间函数表热交换内核。Rust 侧 `HotRuntime`:跨多次
  `step` 调用存活,`call(name, args)` 线程化状态,`hot_swap(program)` 原子替换函数表(并
  重算 liveness 的 `movable_loads`)。自动化测试:旧 `step` 累积状态几轮 → 注入改写后的
  `step` → swap → 断言新版接同一 state 跑出新逻辑。
- **Slice 2**:driver / 文件 watch——`zeta serve <file>`(或语言层 `service`)循环驱动
  `step`,检测源变更并 swap。
- **Slice 3**:语言层语法糖(`reloadable fn` / `service { init; step }`),把"粗粒度替换
  边界"变成显式语言构造,供 native 后端识别并据约束(§3)分别编译 reload / release 两态。
- **(正交)native 后端**:才是"媲美 C/C++"的载体;按 §3 约束把非 reloadable 路径全静态化。

## 5. 验证武器(沿用项目方法论)

每个切片用**自动化测试 + 手算 oracle** 守正确性:状态保持(swap 前后同一 state)、热替换
生效(新逻辑)、不回归(无 swap 时序列不同)。性能约束由 native 后端切片落地时,以"reload
路径外的调用点产出静态分发/内联代码"的断言守护。
