Zeta MVP Baseline 与冻结边界

状态：draft

更新时间：2026-05-28

适用范围：Zeta Stage 0 到 Stage 1 的语言、编译器、运行时、平台 target 和 AI 原生能力取舍。

验收标准：新任务必须能标记为 MVP included、MVP support 或 deferred；README、语言定位、自举路线、target matrix 和 AI capability 文档必须引用同一条 baseline。

Baseline 原则

  • MVP 目标是建立可运行、可测试、可自举演进的最小语言与工具链，不追求完整语言生态。
  • MVP 必须同时保护低门槛和高上限：基础语法要容易学习，编译器和 runtime 边界要能承载系统级、嵌入式和硬件软件演进。
  • release 产物体积是核心质量指标；未使用的 runtime、std、AI 能力、调试元数据和 adapter 不得默认进入最终二进制。
  • 所有 included 项必须服务于 Stage 0 编译器、脚本运行、AOT smoke 或 capability-first 安全模型。
  • 所有 deferred 项不得阻塞 Stage 0 parser、typechecker、MIR、interpreter、WASM/WASI smoke 和 LLVM native smoke。
  • 跨模块能力必须先冻结 schema 和诊断格式，再进入 runtime、tool 或 agent 实现。

MVP Included

领域 Included 冻结条件
语言核心 基础 token、模块、import/export、函数、块表达式、let、if/else、while、return、match、struct、enum、Option、Result、基础整数、Bool、String、数组或 slice。 语法子集必须易学、显式且稳定；AST 节点、source span、错误恢复和 golden dump 格式稳定。
类型系统 静态类型、局部类型推导、函数签名、结构体字段、枚举变体、基础泛型、显式 Result 错误路径。 name resolution 与 typecheck 分层；诊断包含错误码和 source span。
编译器管线 Lexer、parser、AST、diagnostics、name resolution、typechecker、HIR、MIR、MIR verifier、backend interface。 每层有 deterministic dump、verifier 或 golden tests；backend 不直接消费 AST。
执行路径 MIR interpreter 作为首个语义基线，WASM/WASI smoke 作为跨平台 smoke，LLVM native 作为 AOT smoke。 同一 corpus 在 interpreter、WASM/WASI 和 LLVM smoke 中行为一致。
标准库分层 core、alloc、std 分层；String、Vec、Map、Set、路径、文件读取、诊断输出。 依赖方向只能是 std -> alloc -> core；no_std target 不反向依赖 std；系统级和硬件侧软件可以只依赖 core/alloc。
产物体积 runtime 裁剪、按需链接、strip/LTO、最小启动 glue、可关闭 trace/debug 元数据。 每个 target 必须能说明默认链接内容；release smoke 记录 artifact size，体积回归需要 review。
平台 target Linux/macOS/Windows native、wasm32-wasi、wasm32-component；embedded no_std build-only。 每个 target 明确 OS、CPU/ABI、runtime 档位、panic 策略、allocator、CI level。
AI 原生能力 capability 声明、schema、tool contract、agent 边界、policy、budget、timeout、trace 契约。 默认无权限；核心只冻结可检查契约，provider SDK、agent 框架和模型适配器通过标准库或第三方模块按需引入。
工具链 CLI、formatter、最小 LSP、测试命令、HTML 文档、DevGame review/done 证据。 每个任务进入 review 前有验证命令、文档链接和风险记录。

MVP Support

这些能力可以作为实现支撑出现，但不得扩大语言表面积或改变 baseline。

  • Rust Stage 0 宿主语言、Cargo 包管理、测试框架和 CI；Zig 暂不进入当前必需依赖。
  • 早期后端验证顺序固定为 MIR interpreter、WASM/WASI smoke、LLVM native smoke，WASM Component backend 在 capability/plugin 场景成熟后再扩展。
  • 用于调试的 AST/HIR/MIR dump、trace dump 和 corpus runner。
  • 仅服务 smoke 的最小 runtime shim、WASI adapter 和 native ABI glue。
  • 面向文档预览的静态服务器、链接检查和 HTML 结构检查。
  • Zig 可以作为未来 Android/iOS native glue、C ABI、allocator、panic 或 cross-linker adapter 的候选工具，但必须通过独立任务重新评估。

Deferred

领域 暂缓内容 原因
语言特性 宏系统、完整 trait specialization、高阶类型、复杂生命周期、生成器、协程。 会放大 parser/typechecker 和自举复杂度。
并发与 async 完整 async runtime、线程池调度、actor 系统。 需要 runtime 与内存模型稳定后再冻结。
内存模型 完整 borrow checker、unsafe 扩展、手写 allocator API。 MVP 先保证默认安全和基础 owned 模式。
平台 移动端完整应用框架、浏览器 DOM 标准库、嵌入式 runtime-std。 首阶段只要求 build/smoke，不承诺完整产品体验。
AI agent 完整多 agent 编排语言、长期记忆、跨模型策略优化。 MVP 先冻结 capability、schema、tool、trace 和 policy contract。

变更流程

  1. 新增能力必须先在 DevGame 创建任务，并声明它属于 included、support 还是 deferred。
  2. 会改变 baseline 的任务必须进入 review，附带影响范围、兼容风险和回滚方案。
  3. 如果能力进入 included，必须同步更新本文件和对应领域文档。
  4. 如果任务只是 support，不能把临时实现写成长期语言承诺。
  5. 如果能力属于 deferred，只能记录设计动机，不进入 Stage 0 实现路径。
