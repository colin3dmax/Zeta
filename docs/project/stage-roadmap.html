Zeta 阶段路线图

状态：accepted

更新时间：2026-06-04

适用范围：Zeta 从当前 Stage 0 到 Stage 3 self-hosting 的阶段目标、交付物和验收标准。

验收标准：读者可以看到每一阶段要做什么、为什么做、何时进入下一阶段。

Roadmap

每个阶段都有可运行产物，而不是只交付设计口号。

Zeta 的路线按“语言核心 -> MIR 语义 -> 数据建模 -> 模块工程 -> backend smoke -> 自举前端”推进。每一阶段都必须能用测试、dump、官网或线上 smoke 验收。

阶段总览

阶段 目标 核心交付 进入下一阶段条件
Stage 0A：可运行前端 建立 lexer/parser/resolver/typecheck/HIR/MIR/interpreter 的最小闭环。 表达式、绑定、函数、控制流、最小数组、MIR interpreter、REPL、Playground。 run corpus 稳定，官网和 CLI 对同一程序行为一致。
Stage 0B：数据建模 让语言能表达编译器自身数据结构。 结构体值、字段访问、枚举变体、match 执行、Option/Result。 AST/HIR/MIR 的核心数据模型可用 Zeta 子集表达。
Stage 0C：模块与 verifier 把单文件原型推进到工程级编译单元。 最小 module graph、跨模块 name resolution、MIR verifier、corpus 分组、诊断 golden。 目录级 zeta check 能验证多文件 module graph；目录级 zeta run 能执行唯一 main 和跨文件 export fn 调用。
Stage 0D：backend smoke 验证 Zeta 语义能离开解释器，在其他 target 上保持一致。 WASM smoke、WASI target build smoke、LLVM native smoke、artifact size 记录。 tools/smoke-wasm.sh 能构建 wasm32-unknown-unknown，验证 Playground exports，并让 struct/enum/match、数组、字符串 byte 扫描和 typed array builder corpus 返回同一结果；WASI target 未安装时必须明确跳过。
Stage 1：Zeta 前端模块 用 Zeta 编写独立前端模块，Rust Stage 0 仍提供 seed 和 backend。 Zeta parser/AST 子模块、golden dump 对齐、Stage 0 编译 Stage 1。 Stage 1 与 Rust Stage 0 在 golden corpus 上输出一致。
Stage 2：自举核心 Zeta 编译器可以编译自身核心前端。 Stage 1 编译 Stage 2，诊断、HIR/MIR dump 和运行结果一致。 多轮自举输出稳定，Stage 0 只作为冷启动 seed。
Stage 3：完整自举 Zeta 编译器可编译完整 Zeta 编译器和标准工具链核心。 稳定 release、跨平台 smoke、标准库分层、能力契约验证。 Stage 0 不再参与日常构建，只保留为 bootstrap artifact。

近期优先级

  1. Stage 0 最小数组：IntArray、StringArray、BoolArray、数组字面量、下标、.len、MIR verifier、runtime 和用户入口；后续再扩展泛型 Vec 或 slice。
  2. String/char/byte 扫描 API：补 lexer/parser 自举所需的字符串长度、字节访问、ASCII 分类和子串能力。
  3. 最小容器 API：在不引入完整泛型系统前，用保守方式提供 token/span/diagnostic 列表所需的追加和读取能力。
  4. Stage 0D backend smoke：让数组、字符串 byte 扫描、typed array builder、struct、enum、match 和 module corpus 在 WASM/WASI/native smoke 上保持一致。
  5. Stage 1 前端模块：用 Zeta 子集编写 lexer/parser/AST dump；当前 testdata/stage1_frontend 已建立可运行种子，先输出稳定 ast_dump_score，后续再和 Rust Stage 0 的 golden AST dump 对齐。
  6. 文件/路径/诊断 IO：补 Stage 1 编译器读取源码、输出诊断和稳定 dump 所需的最小系统接口。
  7. Stage 1 到 Stage 2 自举：让 Stage 0 编译 Zeta 前端，再用该前端推进 Stage 2。
  8. 多轮 bootstrap 稳定性：至少两轮产物和 golden 输出一致后，才能声明进入可自编译核心。

每阶段共同验收

  • tools/verify.sh 必须通过。
  • 新增语言能力必须同步用户文档、教程、REPL、VS Code 和官网 Playground。
  • 新增可运行语义必须有 run corpus 或 golden dump。
  • 新增用户可见诊断必须有诊断 code/span 单元测试；关键 CLI 诊断还需要有 .stderr golden。
  • 官网发布后必须记录 wasm hash 和线上 smoke 结果。
  • DevGame 任务必须记录决策、测试、提交和发布证据。
