Zeta 阶段路线图

状态：accepted

更新时间：2026-05-30

适用范围：Zeta 从当前 Stage 0 到 Stage 3 self-hosting 的阶段目标、交付物和验收标准。

验收标准：读者可以看到每一阶段要做什么、为什么做、何时进入下一阶段。

Roadmap

每个阶段都有可运行产物，而不是只交付设计口号。

Zeta 的路线按“语言核心 -> MIR 语义 -> 数据建模 -> 模块工程 -> backend smoke -> 自举前端”推进。每一阶段都必须能用测试、dump、官网或线上 smoke 验收。

阶段总览

阶段 目标 核心交付 进入下一阶段条件
Stage 0A：可运行前端 建立 lexer/parser/resolver/typecheck/HIR/MIR/interpreter 的最小闭环。 表达式、绑定、函数、控制流、MIR interpreter、REPL、Playground。 run corpus 稳定，官网和 CLI 对同一程序行为一致。
Stage 0B：数据建模 让语言能表达编译器自身数据结构。 结构体值、字段访问、枚举变体、match 执行、Option/Result。 AST/HIR/MIR 的核心数据模型可用 Zeta 子集表达。
Stage 0C：模块与 verifier 把单文件原型推进到工程级编译单元。 最小 module graph、跨模块 name resolution、MIR verifier、corpus 分组、诊断 golden。 目录级 zeta check 能验证多文件 module graph；目录级 zeta run 能执行唯一 main 和跨文件 export fn 调用。
Stage 0D：backend smoke 验证 Zeta 语义能离开解释器，在其他 target 上保持一致。 WASM smoke、WASI target build smoke、LLVM native smoke、artifact size 记录。 tools/smoke-wasm.sh 能构建 wasm32-unknown-unknown，验证 Playground exports，并让 struct/enum/match corpus 返回同一结果；WASI target 未安装时必须明确跳过。
Stage 1：Zeta 前端模块 用 Zeta 编写独立前端模块，Rust Stage 0 仍提供 seed 和 backend。 Zeta parser/AST 子模块、golden dump 对齐、Stage 0 编译 Stage 1。 Stage 1 与 Rust Stage 0 在 golden corpus 上输出一致。
Stage 2：自举核心 Zeta 编译器可以编译自身核心前端。 Stage 1 编译 Stage 2，诊断、HIR/MIR dump 和运行结果一致。 多轮自举输出稳定，Stage 0 只作为冷启动 seed。
Stage 3：完整自举 Zeta 编译器可编译完整 Zeta 编译器和标准工具链核心。 稳定 release、跨平台 smoke、标准库分层、能力契约验证。 Stage 0 不再参与日常构建，只保留为 bootstrap artifact。

近期优先级

  1. 结构体值语义：结构体字面量、字段访问、字段类型检查、HIR/MIR dump 和 interpreter。
  2. 枚举与 match 执行：枚举变体值、简单模式匹配、MIR lower 和 runtime 行为。
  3. Option / Result 最小模型：显式错误路径和自举编译器所需的基础代数数据类型。
  4. MIR verifier：已接入最小检查，覆盖 local 引用、调用参数、条件类型、return 类型、enum variant、循环内 break/continue 和非 Unit 函数返回路径；下一步扩展完整 CFG 和初始化路径。
  5. 跨模块解析：目录级最小 module/import 图、跨文件 export fn 调用、多文件 run corpus、demo.math.answer() 限定调用、import demo.math as math; 别名调用、export import 函数/结构体/枚举重导出、短名冲突和类型重导出冲突诊断已进入；当前 symbols-dump 可输出稳定导出符号表，下一步继续补完整 backend mangling。
  6. 诊断 golden corpus：CLI 用户可见诊断需要进入 testdata/diagnostics，覆盖单文件和目录级 module graph 错误；当前已覆盖未知名字、类型不匹配、导入冲突和 match 穷尽性，typecheck 也会拒绝未知声明类型名，后续继续把关键 CLI 输出固化为 golden。
  7. WASM/WASI smoke：让同一 corpus 在解释器和 WASI 执行路径上输出一致。

每阶段共同验收

  • tools/verify.sh 必须通过。
  • 新增语言能力必须同步用户文档、教程、REPL、VS Code 和官网 Playground。
  • 新增可运行语义必须有 run corpus 或 golden dump。
  • 新增用户可见诊断必须有诊断 code/span 单元测试；关键 CLI 诊断还需要有 .stderr golden。
  • 官网发布后必须记录 wasm hash 和线上 smoke 结果。
  • DevGame 任务必须记录决策、测试、提交和发布证据。
