Zeta 语言设计过程

状态：accepted

更新时间：2026-05-30

适用范围：Zeta 语言特性从想法、规格、实现、测试到官网发布的完整设计流程。

验收标准：读者可以判断一个新语言特性为什么进入或暂缓，并能看到它如何同步到工具链和官网。

Language Design

语言不是语法集合，而是一组可验证的工程承诺。

每个 Zeta 特性都要回答三个问题：它解决什么真实问题；它是否属于 MVP 或自举路径；它能否被 parser、typechecker、HIR/MIR、runtime、文档和 Playground 同时验证。

设计输入

  • MVP Baseline：决定能力属于 included、support 还是 deferred。
  • 编译器自举路线：决定 Stage 0 需要哪些语义才能表达编译器核心。
  • 语言生态分析：对比 JavaScript、Python、Go、Rust、Swift、Kotlin、Julia 和 WebAssembly 生态的取舍。
  • target matrix：约束 runtime、panic、allocator、WASM/WASI/component 和 embedded 能力。
  • AI capability 契约：约束权限、schema、tool、agent、trace 和默认最小授权。

特性进入流程

  1. 动机：说明用户、编译器或 runtime 为什么需要该能力。
  2. 归类：标记为 MVP included、MVP support 或 deferred；deferred 不进入 Stage 0 parser 接受路径。
  3. 语法草案：写出最小示例和错误示例，避免只描述抽象概念。
  4. 语义边界：定义类型规则、运行时行为、诊断错误码和 source span。
  5. IR 形态：决定 AST/HIR/MIR 如何表达该特性，dump 必须稳定。
  6. 实现与测试：先补 corpus/golden，再改 lexer、parser、resolver、typecheck、runtime 或 backend。
  7. 工具同步：REPL、VS Code、官网 Playground、用户文档、教程和 API 文案同步。
  8. 发布验收：运行 tools/verify.sh，必要时 tools/verify.sh --deploy --live，并记录 DevGame 证据。

当前语言层次

层次 已实现 下一步
标量与表达式 Int、String、Bool、整数算术、比较、布尔逻辑。 完善字符串操作和基础错误路径。
局部状态 let、let mut、赋值、局部类型注解。 明确初始化状态和 MIR verifier 规则。
控制流 if/else、while、return。 让 match lower 到 MIR 并可执行。
函数 函数声明、参数、返回类型、函数调用、无参数 main 执行。 跨模块函数符号和导入解析。
数据建模 struct、enum 的 parse / AST / HIR dump。 结构体值、字段访问、枚举变体、Option/Result。
工程化 CLI、REPL、VS Code、官网 Playground、HTML 文档和线上 smoke。 CI 化连续验证，增加 corpus 分组和 backend smoke。

设计时避免的问题

  • 不因为语法好看就加入核心；必须证明它服务 MVP 或自举。
  • 不让 backend 直接消费 AST；backend 必须通过 MIR 或稳定接口。
  • 不在 parser 里接受 deferred 语法，以免未来兼容成本过高。
  • 不把官网演示写成模拟器；线上用户看到的必须是真实编译器结果。
